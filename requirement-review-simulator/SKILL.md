# 产品经理需求评审模拟器

## 核心价值定位
- **不是文档生成工具，而是评审前的攻防推演模拟器**
- 引入五大角色的真实人设与口头禅，模拟真实职场的压迫感
- 生成可分享的"PRD 存活率体检报告"，具备社交传播属性
- 补齐行业空白：跨部门协作冲突预判 + 决策推演闭环

## 适用场景
- 产品经理在需求评审前预演，担心被技术/运营/老板等角色挑战
- 需要多方博弈下的可落地决策方案，不只要"问题列表"
- 需要会议可执行资产：脚本、分工表、决策闸门、风险预案
- 想看看自己的 PRD 能在"地狱模式"活几分钟（情绪价值）

## 不适合
- 从零撰写 PRD（这是压测器，不是写作工具）
- 已上线产品的全面复盘（支持局部迭代评审，不做系统性事后诊断）
- 替你做最终决策（给 Go / Conditional Go / No Go 建议，拍板仍由你）

## 输入层

### 交互流程：引导采集 → 确认/跳过 → 生成报告

用户说了需求后，**不要直接生成报告**，而是先输出一份「信息采集清单」引导用户补充。

**第一步：用户说了需求** → **第二步：技能输出采集清单** → **第三步：等待用户回复** → **第四步：生成 HTML 报告**

信息采集清单格式：

```
收到！在正式开始评审推演之前，我先帮你梳理一下关键信息。
已有的信息我帮你预填了，你可以逐项确认或修改，也可以直接说「跳过，直接生成」。

━━━━━━━━━━━━━━━━━━━━━━
📋 信息采集清单
━━━━━━━━━━━━━━━━━━━━━━

1️⃣ 需求名称：[已提取 / 待补充]
2️⃣ 业务目标：[已提取 / 待补充]
3️⃣ 核心功能：[已提取 / 待补充]
4️⃣ 评审残酷度：
   🟢 新手村（温柔建议）
   🟡 实战（标准大厂流程，默认）
   🔴 地狱（全员恶人，高压质疑）
   → 当前预设：[根据用户语气推断]
5️⃣ 行业：[已推断 / 待补充]
6️⃣ 公司规模：[已提取 / 待补充]
7️⃣ 技术栈：[已提取 / 待补充]

🔟 需求类型：
   📦 大型产品功能 🔧 工具类/内部工具 🚀 MVP / 验证期 🔄 迭代优化 ⚖️ 合规敏感型
   → 当前预设：[根据需求内容推断]
1️⃣1️⃣ 维度适用性：自动识别是否存在 N/A 维度
1️⃣2️⃣ PRD 来源：
   📄 用户自有 PRD / 🌐 公开/展示版 PRD
   → 当前预设：[根据输入方式推断]

💡 信息越详细，报告越精准。也可以直接回复「跳过，直接生成」。
```

用户回复处理：逐项补充→更新后生成 / "跳过"→用推断值生成 / 只补部分→混合使用

### 推断逻辑

| 信息 | 推断方式 | 推断失败时 |
|------|----------|-----------|
| 评审残酷度 | "地狱/压测/最难"→🔴；"简单看看"→🟢 | 预填 🟡 实战 |
| 行业 | 从需求关键词推断（拼团→电商；投顾→金融） | 标"待补充" |
| 公司规模 | 用户提到人数或阶段时提取 | 标"待补充" |
| 技术栈 | 用户提到具体技术时提取 | 标"待补充" |
| 约束条件 | 从需求中提取时间、预算、人力 | 标"待补充" |
| 需求类型 | 从需求特征推断 | 默认"大型产品功能" |
| PRD 来源 | 贴 URL/文章→🌐；说"我的PRD"→📄 | 预填 📄 用户自有 |

### HTML 报告标签规则
- 顶部标签区只显示**有确认信息的标签**，推断值后加 `[推断]` 标记
- 决策结论标签和 PRD 来源标签必须始终显示
- 展示版 PRD 在评分卡区域增加说明：`已启用展示版评分校准`

## 处理层

按顺序执行，详细规则见各 references 文件，不在此重复：

1. **信息采集与确认** — 提取已有信息，推断可推断项，输出采集清单；用户确认/跳过后整理最终信息
2. **需求类型识别** — 判断类型，加载对应权重（五维基准均 20%，按类型调整）；维度可判 N/A，`逻辑自洽` 永不 N/A；详见 scoring-engine-deterministic.md
3. **PRD 来源校准** — 📄 自有：严格标准；🌐 展示版：12 个豁免子项强制 2 分，禁止追问必然缺失的内部信息；详见 scoring-engine-deterministic.md
4. **加载角色人设** — 按残酷度调整五大角色语气/问题数/黑话浓度/预设立场；详见 review-playbook.md Part I

   | 残酷度 | 语气 | 每角色问题数 | 预设立场 |
   |--------|------|------------|---------|
   | 🟢 新手村 | 温和鼓励 | 5–8 | 基本支持 |
   | 🟡 实战 | 专业直接 | 10–12 | 中立审视 |
   | 🔴 地狱 | 尖锐带刺 | 12–15 | 默认反对 |

5. **逐角色攻防推演** — 五角色依次输出：质疑 + 真实关切 + 严重等级；N/A 维度对应问题簇降权或移除
6. **跨部门协作推演** — 识别依赖/冲突关系，生成 RACI 矩阵；详见 review-playbook.md
7. **决策推演** — A/B/C 三方案 + 四闸门判定 → Go / Conditional Go / No Go + T+7/T+30 回看节点
8. **应对话术** — 为致命/重要级质疑生成 `[认同关切]+[数据支撑]+[逻辑推演]+[备选方案]`；挑 3 题生成"神回复"；详见 review-playbook.md Part II

## 输出层（爆款四件套）

### 模块一：PRD 存活率体检报告

```
🏥 PRD 存活率体检报告

需求名称：[XXX]
评审模式：[🟢新手村 / 🟡实战 / 🔴地狱]
综合存活率：[0-100 分]（低于 60 分建议重写）

┌─────────┬──────────┬──────────────────────────┐
│  维度    │ 得分(0-10)│ 毒舌评语                  │
├─────────┼──────────┼──────────────────────────┤
│ 逻辑自洽 │          │                          │
│ 技术友好 │          │                          │
│ 运营价值 │          │                          │
│ 老板满意 │          │                          │
│ 合规安全 │          │                          │
└─────────┴──────────┴──────────────────────────┘

☠️ 致命死穴：[最可能导致项目流产的一个问题]
🛡️ 救命稻草：[能挽回局面的核心论据]
```

**评分必须严格遵循** [references/scoring-engine-deterministic.md](references/scoring-engine-deterministic.md) 中的确定性评分引擎规则（含展示版子项评分指令和硬校验规则）。

**⚠️ 综合存活率计算（强制 9 步）**：
1. 识别 PRD 来源（📄 自有 / 🌐 展示版）
2. 识别需求类型
3. 判定 N/A 维度
4. 逐子项评分
5. 子项汇总得维度分
6. 一致性校验 + 硬校验
7. 归一化权重
8. 公式计算综合分
9. 映射评级与决策结论

> **单一事实源（SSOT）**：权重参数、子项清单、分档阈值、展示版豁免与硬校验细则，均以 `references/scoring-engine-deterministic.md` 为唯一准则；`SKILL.md` 仅保留执行顺序，不重复维护具体参数，避免规则漂移。

### 模块二：五角色质疑清单

每角色格式：`## [角色名]（[口头禅]）` + 编号表格（质疑/等级/真实关切）

### 模块三：职场救命神回复

从所有质疑中挑**最难回答的 3 个**，每个提供：致命问题 → 普通回复 → 神回复 → 技巧解析

### 模块四：标准会议资产

1. **需求优化建议**：按 必须修复 / 可选优化 / 延后讨论 分级
2. **评审会议脚本**：开场白(3min) → 核心论证(10min) → 风险预案(5min) → 收尾(5min)
3. **跨部门协作包**：RACI + 关键依赖 + 冲突化解
4. **决策推演包**：A/B/C 方案对比 + Go 结论 + 回看指标

### 输出格式规范

**默认输出 HTML 报告**，严格按照 [references/report-template-pro.html](references/report-template-pro.html) 的结构和样式生成。报告 11 大分区（顺序不可打乱）：Hero → Radial Hub（仪表盘 + 雷达图）→ Callouts（Fatal Flaw / Lifeline）→ Decision Gates + A/B/C Plans → Five-Role Challenge Board → Killer Replies TOP 3 → Cross-Team Collaboration（RACI + 冲突化解）→ Review Meeting Script → Optimization Suggestions → Action Checklist → Disclaimer。

**输出格式锁（强制）**：
- 分区数量、分区顺序、分区标题保持与模板一致，禁止新增/删减/重排
- 顶部信息条必须包含：评审模式、PRD 来源、需求类型、决策结论（其余标签按信息完整度显示）
- 评分卡必须包含：五维得分、综合存活率、评级、致命死穴、救命稻草
- 决策闸门必须包含：价值/风险/资源/战略四闸门与最终结论

**UI 风格**：浅蓝渐变 Hero（且慢色调）+ 白色卡片 + SVG 仪表盘/雷达图，与 opc-board 共享同一设计语言

**评分明细展示**：每维度 KPI 卡片可展开子项明细，展示版豁免子项标记 `🏷️展示版豁免`，底部显示一致性校验结果。

**去AI化文风**：禁止空泛套话，结论必须具名、具数、具动作。

## 质量标准

- **真实感**：地狱模式必须包含行业黑话，质疑符合角色人设
- **犀利度**：评分卡评语"扎心"但客观，不能全是好话
- **可用性**：神回复做到"怼回去"且"不翻脸"
- **完整性**：必须有决策结论（Go/Conditional Go/No Go）
- **层次感**：应对话术包含数据支撑 + 逻辑推演 + 备选方案三层
- **可执行性**：建议绑定影响维度（成本、周期、收益、风险）
- **跨部门视角**：显式体现部门间冲突与化解动作

## 注意事项

- **始终先输出信息采集清单**，用户说"跳过"才用推断值
- **保持"对抗但不对立"**：问题尖锐，表达建设性
- **评分必须遵循确定性引擎**：逐子项打分 → 公式计算，禁止凭感觉（细则见 scoring-engine-deterministic.md）
- **来源模式前置识别**：评分前先判断 📄自有 / 🌐展示版（校准规则见 scoring-engine-deterministic.md）
- **残酷度一旦确认不中途降级**
- **评分卡必须有区分度**：不允许人为拉平

### 生成前一致性自检（强制）

输出 HTML 前逐项核对，不通过先修正：
1. 公式一致：`composite_score = Σ(维度得分 × 有效权重) × 10`
2. 评级一致：综合分与 S/A/B/C 分档匹配
3. 结论一致：评级与 Go / Conditional Go / No Go 映射匹配
4. 展示版一致：豁免子项 = 2，一致性校验只上调
5. N/A 一致：裁剪维度不高频追问，权重已归一化
6. 格式一致：9 大分区齐全且顺序不变
